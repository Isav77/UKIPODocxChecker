About this tool

What it checks

This tool inspects the XML inside a Microsoft Word DOCX file against the points of the One IPO Directions that can be checked structurally. Each finding is grouped by direction number for traceability.

  • Direction 10 — DOCX format
  • Direction 11 — No alt-text on images
  • Direction 13 — Sequence listings must be ST.26 XML (heuristic detection)
  • Direction 15 — Application Documents submitted separately (heuristic)
  • Direction 18 — No OLE objects
  • Direction 19 — No comments
  • Direction 20 — No tracked changes or annotations
  • Direction 21 — No version histories (heuristic: customXml parts, settings flags, body text patterns)
  • Direction 22 — URL extraction for manual verification
  • Direction 24 — Minimum 10-point font; full list of fonts in use for manual review (with known condensed/narrow and symbol/decorative fonts flagged)
  • Direction 25 — Description Document must not include a title (heuristic)
  • Direction 23 ? Schedule 2 paragraphs 3, 4, 6, 7, 9 — No frames; page numbering consecutive in a single series; page numbers at top or bottom, centred, not in margin; minimum 20 mm margins; minimum 1.5 line spacing
  • Document Inspector parity — Hidden text, near-invisible text colour, decorative or review-style formatting (strikethrough, highlighting, shading), macros, embedded code, document properties metadata, page watermarks, footnotes and endnotes, numbering anomalies, embedded image metadata (EXIF, XMP, PNG text chunks)

What it does not fully check

Direction 17 requires the document to be wholly and completely legible. That is partly addressed (small font, near-white text, symbol fonts are surfaced), but legibility has many other dimensions the tool cannot judge: text rendered as a flattened image, low-contrast colour combinations, hand-drawn or photographed sections, or unusual fonts that aren't on any known list. The font list (Direction 24 check) is presented in full to let the reviewer catch the rest manually.

Page numbers

Page numbers are derived from page-break markers Word writes the last time it renders the document (w:lastRenderedPageBreak) and any explicit user-inserted page breaks. If a document has never been opened in Word (for example, one generated programmatically), these markers will be absent and locations will simply say "Body" rather than "Page N". Open the document once in Word and save it to populate the markers.

What it does not check

  • Direction 12, 26 — Drawings (PDF). Out of scope for v0.3.
  • Direction 13 (ST.26 compliance) — The check above flags likely sequence content, but does not validate ST.26 conformance. Use the WIPO Sequence tool for that.
  • Direction 17 — Wholly and completely legible. Subjective; the tool cannot give a definitive answer.
  • ODT format. Different XML schema. Limitation only because DOCX dominates today. Save ODT files as .docx via Word before running this tool.

Known limitations

DOCX formatting is layered: theme defaults, linked styles, paragraph properties and run properties combine to produce what Word actually renders. This version resolves style inheritance via the basedOn chain for font size, hidden text and text colour, including theme colour references. Sections of the document still not covered include: text rendered as a flattened image, low-contrast colour combinations the threshold doesn't catch, hand-drawn or photographed sections, and content whose effective formatting depends on conditional logic (e.g. content controls) that the static XML scan cannot evaluate. Always confirm by visual inspection in Word before filing.

How findings are reported

Each finding shows the direction number, the offending value and a short text excerpt for location. Excerpts are limited to 60 characters and never leave this tab. Findings flagged as "fail" represent requirements where the relevant Direction or Schedule 2 paragraph uses "must" and the check is deterministic (margins, line spacing, font size, hidden text, tracked changes, comments, OLE objects, alt-text and others). Findings flagged as "warn" indicate items that need manual review for one of two reasons: either the check is heuristic and could be a false positive (Direction 13 sequence-listing patterns, Direction 15 combined-document detection, Direction 21 version-history indicators, Direction 25 first-paragraph title detection), or the item is not directly prohibited by the Directions but typically signals cleanup work (decorative formatting, URLs for completeness review). "Info" indicates a check whose conclusion is limited by what is structurally observable.

Privacy and connectivity

The tool processes documents entirely in your browser. Nothing is uploaded anywhere. When you select a file (whether by drop or by clicking the upload zone), the browser reads its bytes into memory using the standard File API, JSZip unpacks the DOCX, and the checks read the unpacked XML — all without any network request. The page is delivered as a single HTML file with the JSZip library inlined; once loaded, it can be used offline.

A strict Content Security Policy enforces this property at the browser level. default-src 'none', connect-src 'none' and form-action 'none' together block any outbound request the page might be coerced into making (whether through code in the page or through a malicious DOCX). img-src data: means images can only be embedded as data URIs, not loaded from a server. The browser will refuse anything not on the allowlist; this is verifiable in any browser's developer tools (Network tab) by confirming zero outbound requests during a run.

This matters for pre-published patent applications: privileged client IP never leaves the device on which the check runs.

For IT and auditors

The tool includes a self-test harness intended for IT or audit use, not for fee earners. Visit this page with #selftest appended to the URL (or follow this link in the browser address bar after opening the file: ...ukipo_checker.html#selftest) to run three embedded reference documents through the full check pipeline and report any deviation from expected results. The self-test also verifies that the inlined JSZip library matches the SHA-512 hash captured when this build was produced. A passing run is evidence that the tool is producing the results it was designed to produce; a failing run indicates the tool has been edited and the change needs investigation.

Version and release notes

The current version is shown in the footer and on the audit panel of each run. Notable changes by version:

  • v0.5.1: Adds a self-test harness for IT and audit. Visiting the page with #selftest appended to the URL runs three embedded reference documents through the full check pipeline and reports a pass or fail per fixture, together with a SHA-512 verification of the inlined JSZip library against the hash captured when the tool was built. The harness is invisible to fee earners during normal use and does not change anything they see.
  • v0.5.0: Hard size limits (100 MB input, 500 MB unpacked) to defend against malformed or maliciously crafted documents that expand massively when unzipped. Adds five new checks covering document properties metadata (author, last modified by, company, manager, template, total edit time, plus any SharePoint or iManage custom properties under docProps/custom.xml), page watermarks (classic Word watermarks via VML and floating shapes in headers carrying typical watermark text), footnotes and endnotes (Word stub separators ignored; real author-added notes counted via body references), numbering anomalies (lists forced to start at a value other than 1 via startOverride), and embedded image metadata (EXIF tags including camera make/model and software, GPS data presence, XMP packets, Photoshop IRB segments, PNG tEXt chunks). The EXIF parser is focused and not exhaustive: TIFF, HEIC and HEIF formats are flagged by extension without parsing; XMP and GPS sub-IFD contents are detected but not decoded. Audit footer now also shows the unpacked file size and compression ratio.
  • v0.4.3: Refined the status assignment for checks that rely on heuristic detection. Where a finding requires reviewer judgement to confirm — Direction 13 (sequence-listing patterns in text), Direction 15 (heading-based detection of combined Application Documents), Direction 21 (version-history indicators including custom XML and heading patterns) and Direction 25 (first-paragraph patterns that may be the invention title or may be a section heading like "TECHNICAL FIELD") — the status is now "warn" rather than "fail". Deterministic checks (margins, line spacing, font size, page number alignment, hidden text, tracked changes, comments, OLE objects, alt-text and others) continue to fail when the Directions use "must".
  • v0.4.2: Plain-English pass across every check. OOXML element names (<w:ins>, <w:trackRevisions>, <w:framePr> and similar) replaced with descriptions a fee earner can act on. Direction 24 letter spacing now reported in points rather than twentieths of a point. "twips" removed from margin findings. Direction 20 tracked-change types named individually (Insertion, Deletion, Move source, table cell changes, etc.). File paths used as location labels ("word/settings.xml", "styles.xml docDefaults") replaced with friendlier labels. Most suggested fixes rewritten with clearer Word menu paths. The font-list "For manual review" entry reframed as "Other fonts in use".
  • v0.4.1: Clarity pass. Word built-in non-body styles (BalloonText, CommentReference and similar) are no longer flagged at the style level for Direction 24 — they exist in every document by default at sub-10pt sizes but don't carry body content. Headers and footers are now named specifically ("Default footer", "First-page header", "Even-page footer") so the fee earner knows which one to edit. "Runs" renamed to "occurrences" throughout. Findings whose content has no visible text now describe what's actually there ("page number field", "image or drawing", "tab character") rather than "(no text)". Suggested fixes can now use numbered steps and the Direction 24 suggested fix has been rewritten as a step-by-step.
  • v0.4.0: Adds an audit footer below the Back to top button showing the SHA-256 hash and tool version, and a Checked-at subtitle under the document name (UK date/time). Adds icons to status pills so the fail/warn/pass/info distinction does not depend on colour alone. Adjusts the muted-text palette to meet WCAG AA contrast. Error messages now include help text directing users to raise an issue if a finding looks wrong.
  • v0.3.0: Promotes eight checks to "fail" status to align with the Directions' use of "must": Direction 13 (sequence listings), Direction 15 (combined Application Documents), Direction 21 (version histories), Direction 24 (condensed/narrow fonts), Direction 25 (no title in Description), Schedule 2 paragraph 3 (no frames), Schedule 2 paragraph 6 (page number position), and Schedule 2 paragraph 9 (line spacing). Resolves style inheritance via the basedOn chain plus docDefaults for the font size, hidden text and text colour checks. Resolves theme colour references against theme1.xml. Extends line spacing detection to cover the exact and atLeast line rules. Direction 15 walks headers and footers and resolves heading styles via the chain. Direction 20 covers a broader set of revision elements. ST.25 tag detection requires whitespace boundaries to reduce false positives. "Back to top" moved to immediately below the results.
  • v0.2.0: Adds Schedule 2 paragraph 4 (page numbering) and paragraph 6 (page number position), Direction 21 (version histories) and Direction 25 (no title in Description). Hardens HTML rendering, prevents prototype pollution from malformed relationship IDs, and clarifies the encryption-check message.
